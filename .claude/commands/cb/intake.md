인자: $ARGUMENTS (선택 — 특정 인테이크 파일 경로. 없으면 루트 INTAKE-*.md 자동 탐지)

docs/GUIDE.md 의 §8 /cb:intake 흐름과 §4.6.3 회수 후 처리 절차를 따른다.

## 절차

1. **인테이크 파일 회수·박제**
   - 인자 없으면 프로젝트 루트의 `INTAKE-*.md` 자동 탐지
   - 여러 개면 목록 보여주고 선택받음
   - 파일을 `docs/10-requirements/intake/` 로 이동·박제
   - 이후 이 사본이 불변 원천

2. **형식 점검 (PM)**
   - C 표의 ID 중복/공백 확인
   - `?` 항목 집계
   - 분석 가능 수준인지 판단

3. **planner 정제 디스패치**
   인테이크 파일을 입력으로 planner 에이전트 호출.
   역할은 **정제** — R-ID 를 FR 로 구조화하고 AC 작성.
   인테이크에 없는 기능 추가 금지.

4. **질문 회신서 (RFI) 작성**
   planner 의 미해결 질문을 `docs/10-requirements/intake/RFI-<회차>.md` 로 정리.
   질문마다 "왜 필요한지 + 답 예시" 포함.
   → 사람에게 전달

5. **루프**
   사람 답변 → planner 반영 → 잔여 질문이 진행 차단급 아닐 때까지 반복.
   차단급 아닌 잔여 질문은 "가정" 으로 명기하고 진행.

6. **추적성 고정**
   정제된 모든 FR 에 원천 R-ID 표기 (R-07 → FR-12 식).
   planner 제안 기능은 `P-` 접두로 구분.

7. **[사람 게이트] 베이스라인 확정**

8. **마일스톤 분할**
   우선순위(M 우선) + 의존관계 + 일정 제약으로 마일스톤 안 2개 내외 제시.
   각 안에 포함 FR, 마일스톤 목표, 리스크 기술.
   → **[사람 게이트] 분할 승인**
   → 보드에 마일스톤·트래킹 이슈 골격 전체 + 첫 마일스톤 백로그 이슈만 생성
      ⚠️ GitHub: 이슈 생성은 3단계 1쌍 — platform-ops §2 gh CLI 블록 참조
