### developer 단위 테스트 충실성 점검 (형식적 테스트 여부)

`tests/test_parser_fr26.py` 22개 테스트를 개별 열람한 결과, 전량 실제 AC의
Then을 실측 단언한다 (길이/타입/좌표값/부분문자열/mso_shape_type 정확값 비교
등). `assert True`류 항상-참 단언, 예외를 삼키고 통과 처리하는 패턴, AC와
무관한 보조 단언만으로 구성된 테스트는 발견되지 않았다. python-pptx mock은
공개 API로 재현 불가한 경우(`paragraph.level is None`, 예외를 던지는
손상 도형, XML 예외 경로)에만 국소적으로 사용되어 ADR-207 취지에 부합한다.
결론: **형식적 테스트 아님 — 실질적 검증**.

QA 검증 과정에서 developer 코드의 실제 동작 중 두 가지를 확인·정정했다
(결함 아님, 이해 정정):
1. `_parse_text`의 `placeholder_format` 접근 시 `ValueError`는 "non-placeholder
   shape"의 정상 신호로 이미 내부에서 흡수된다(FR-04 기존 설계). 손상 도형을
   흉내낼 때는 그 외의 예외 타입(RuntimeError 등)을 써야 ADR-204 강등 경로가
   실제로 발동한다 — developer의 AC7 테스트는 이미 올바르게 RuntimeError를
   사용했음을 재확인했다.
2. `presentationml/2006/ole` 같은 표준 OLE 관계 uri는 python-pptx가 자체적으로
   `shape_type`(예: LINKED_OLE_OBJECT)을 인식하므로 `_graphic_frame_type_label`
   폴백 경로(1차가 아닌 2차 라벨링)를 타지 않는다. AC6의 "UNKNOWN이 아닌 실제
   라벨" 요구는 어느 경로를 타든 충족된다.
