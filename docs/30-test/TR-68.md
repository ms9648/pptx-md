### developer 단위 테스트 충실성 점검 (형식적 테스트 여부)

`tests/test_description_pipeline.py`(FR-27 관련 신규분, AC2/AC3/AC4~AC10),
`tests/test_api.py`(TestFr27ConvertOptions·TestAc1Fr27EndToEndDescriber·
TestAc8Fr27DeterministicGoldenRegression), `tests/test_mermaid.py`(AC4/AC5
단위)를 개별 열람했다. 전량 실제 AC의 Then을 실측 단언한다:

- AC7: `ConcurrencyCountingDescriber`로 `max_observed<=N`을 직접 단언(항상 참
  단언 아님, `time.sleep`으로 실제 겹침을 유도).
- AC6: `describer.calls` 길이 등식(고유수) + 두 위치 description 동일성
  이중 단언.
- AC8: `max_workers=1` vs `=8` 최종 매핑 완전 동일 비교, `convert()` 2회
  바이트 동일(`encode()` 비교).
- AC9: `assemble_document`까지 연결해 FR-22 마커 문자열(`"슬라이드 1 이미지"`)
  실제 등장을 확인(문자열 포함 여부 미검증 아님).

`assert True`류 항상-참 단언, 예외를 삼키고 통과 처리하는 패턴, AC와 무관한
보조 단언만으로 구성된 테스트는 발견되지 않았다. mock 사용(FakeDescriber류)은
이슈가 명시한 "VLM 실제 API 호출 금지" 제약에 따른 정당한 사용이다.

QA 검증 과정에서 확인·정정한 사항(결함 아님, 이해 정정):
1. `description="", not None` 이면 `_render_image`가 "설명 있음" 경로를 타
   FR-22 위치 마커가 나오지 않는다 — FR-22 마커를 유발하려면 반드시
   `description is None`이어야 한다. 이는 `describe()` 예외뿐 아니라
   EMF/WMF 벡터 변환 실패(`vector.convert_vector_to_png`→None) 경로로도
   동일하게 발생함을 TC-68-26으로 독립 재현·확인했다(developer는 예외
   경로만 검증).
2. `hash()` 내장 함수는 프로세스 간(재시작 시) 값이 달라질 수 있으나(해시
   랜덤화), 동일 프로세스 내에서는 동일 객체에 대해 일관되므로 TC-68-01의
   토큰 비교(같은 pytest 프로세스 내에서 생성·검증)에는 영향이 없음을
   확인했다. AC8 골든 비교 테스트(TC-68-22~24)는 `hash()`에 의존하지 않고
   `sum()`/`repr()` 기반 결정적 함수만 사용해 이 리스크를 원천 배제했다.

결론: **형식적 테스트 아님 — 실질적 검증**.
